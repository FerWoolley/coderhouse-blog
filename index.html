Desafíos en Producción: Depurando CORS y Sesiones en el Despliegue de un Stack MERN

CH

Estudiante Coderhouse

Diplomatura de Desarrollo Full Stack

Contexto

Durante el desarrollo de nuestro proyecto final para la Diplomatura, implementamos una aplicación de gestión de tareas colaborativas utilizando el clásico stack MERN (MongoDB, Express, React y Node.js). En el entorno de desarrollo local (localhost), todo funcionaba a la perfección: el inicio de sesión autenticaba al usuario, guardaba una cookie de sesión cifrada y permitía consumir las rutas protegidas de nuestra API.

El reto comenzó cuando llegó el momento de desplegar la aplicación a producción. Decidimos hospedar el frontend de React en Vercel (bajo el dominio https://gestor-frontend.vercel.app) y la API de Express en Render (bajo el dominio https://gestor-api.onrender.com).

El Problema

Inmediatamente después de realizar los despliegues, nos enfrentamos a un muro: los usuarios no podían iniciar sesión. Al inspeccionar la consola de las herramientas de desarrollo del navegador, nos topamos con un error persistente y temido: 

Access to XMLHttpRequest at 'https://gestor-api.onrender.com/api/auth/login' from origin 'https://gestor-frontend.vercel.app' has been blocked by CORS policy: Response to preflight request doesn't pass access control check: No 'Access-Control-Allow-Origin' header is present on the requested resource.

Tras habilitar CORS en el backend de forma básica permitiendo el origen del frontend, logramos que la petición de login respondiera con éxito (200 OK). Sin embargo, surgió un segundo problema silencioso: las cookies de sesión (JSON Web Tokens en cookies seguras) no se guardaban en el navegador del frontend. Como consecuencia, cualquier petición posterior a rutas protegidas devolvía un código de error de autenticación 401 Unauthorized.

⚠️ Impacto en el equipo

El error detuvo las pruebas de usabilidad y generó cierta frustración y tensión en el equipo, ya que faltaban pocas horas para la presentación del prototipo inicial y el sistema parecía "completamente roto" en producción a pesar de funcionar en nuestras máquinas.

Acciones Tomadas

Adoptando una mentalidad de crecimiento, entendimos que este cuello de botella era una oportunidad de aprendizaje profundo sobre los protocolos de seguridad web en producción y cómo difieren del desarrollo local. Estructuramos un plan de acción constructivo:

1. Post-Mortem Constructivo y Diagnóstico

Nos reunimos a revisar el flujo de red. Identificamos que el problema de CORS original se debía a que el middleware en el servidor no estaba configurado para aceptar peticiones con credenciales (cookies) desde orígenes cruzados.

En el entorno local, al estar ambos servicios bajo localhost, la seguridad de las cookies es mucho más permisiva. En producción, al haber dominios cruzados (vercel.app y onrender.com), los navegadores modernos bloquean por defecto las cookies a menos que se cumplan criterios específicos de seguridad (directivas SameSite y Secure).

2. Modificaciones en el Backend (Express.js)

Ajustamos la configuración del middleware de CORS para especificar explícitamente el origen del frontend y permitir el intercambio de credenciales: 

// server.js
const cors = require('cors');

app.use(cors({
    origin: 'https://gestor-frontend.vercel.app',
    credentials: true, // Permite el envío de cookies a través de dominios
    methods: ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS'],
    allowedHeaders: ['Content-Type', 'Authorization']
}));

Adicionalmente, modificamos la lógica que genera y firma la cookie de sesión en el controlador de inicio de sesión: 

// controllers/authController.js
res.cookie('token', token, {
    httpOnly: true, // Previene acceso por scripts maliciosos (XSS)
    secure: true,   // Obligatorio para SameSite=None, requiere HTTPS
    sameSite: 'none', // Permite que la cookie se envíe en peticiones cross-site
    maxAge: 24 * 60 * 60 * 1000 // 1 día
});

3. Modificaciones en el Frontend (React y Axios)

En el cliente, configuramos una instancia global de Axios para asegurar que todas las peticiones asíncronas incluyeran de manera automática el parámetro withCredentials: 

// api/axiosInstance.js
import axios from 'axios';

const api = axios.create({
    baseURL: 'https://gestor-api.onrender.com/api',
    withCredentials: true // Adjunta cookies de sesión automáticamente
});

export default api;
💡 La Solución Definitiva

Una vez aplicados estos cambios en ambos repositorios, el flujo de autenticación funcionó perfectamente. La cookie se almacenó con seguridad en el navegador y las llamadas API a los módulos protegidos fueron exitosas.

Aprendizajes

  • Diferencias de Entorno: Aprendimos que "funciona en mi máquina local" no es garantía de éxito en producción. Los navegadores aplican políticas de seguridad mucho más estrictas para dominios reales que para localhost.
  • Seguridad Web Avanzada: Comprendimos el funcionamiento técnico detrás del protocolo de Preflight (OPTIONS) de CORS y la anatomía de las cookies HTTP-Only con sus configuraciones de privacidad (SameSite=None y Secure).
  • Planificación de Despliegues: De ahora en adelante, definiremos políticas de CORS e integraciones de red desde la primera fase de desarrollo del proyecto para evitar cuellos de botella de última hora en el despliegue.

Reflexión sobre Feedback Radicalmente Sincero

Durante este proceso, la aplicación de la metodología de Feedback Radicalmente Sincero (Radical Candor) de Kim Scott fue crucial para el éxito del equipo. Cuando surgió el fallo en el despliegue, estuvimos al borde de culpar al miembro del equipo encargado de DevOps.

En su lugar, decidimos actuar desde el cuadrante del Afecto Personal con Desafío Directo. Tuvimos una conversación muy abierta y honesta: le manifestamos directamente que la app estaba fallando en producción y que necesitábamos resolverlo con urgencia (desafío directo), pero al mismo tiempo nos pusimos a disposición para investigar juntos, recordándole que valorábamos mucho su esfuerzo y que sabíamos que las configuraciones de hosting son complejas (afecto personal).

Esto evitó que cayéramos en la insinceridad manipuladora (no decirle nada y arreglarlo a sus espaldas) o en la agresión ofensiva (criticar su trabajo rudamente frente a todos). Como resultado, el compañero no se sintió atacado, sino apoyado, y pudimos encontrar la solución en menos de una hora, fortaleciendo la confianza del grupo.
